# GEDCOM concepts weÔÇÖll explicitly model (subset)

The dummy examples and the Agda recipes below are grounded in these GEDCOM v7 ideas:

- A GEDCOM dataset is a header, followed by 0+ records, ending with a trailer.The FamilySearch GEDCOM SpecifiÔÇŽ
- Pointers are either a real cross-reference pointer or the special`@VOID@`placeholder (ÔÇťvoid pointerÔÇŁ).The FamilySearch GEDCOM SpecifiÔÇŽ
  - `@VOID@`is used when data is removed but structure/pointers must remain (export/round-trip friendliness).The FamilySearch GEDCOM SpecifiÔÇŽ
  - `@VOID@`can also appear as a placeholder for an unknown child in a`FAM`record.

- The core linkage is between`INDI`(individual) and`FAM`(family) records, and linkages are expected to be consistent in both directions:
  - `FAM`uses`HUSB`,`WIFE`,`CHIL`pointers into`INDI`.The FamilySearch GEDCOM SpecifiÔÇŽ
  - The pointed-to`INDI`records should point back using`FAMS`(as spouse) and`FAMC`(as child).The FamilySearch GEDCOM SpecifiÔÇŽ

- Pedigree / child-family status uses`PEDI`(enum:`ADOPTED`,`BIRTH`,`FOSTER`,`SEALING`,`OTHER`with a`PHRASE`).The FamilySearch GEDCOM SpecifiÔÇŽ
- Adoption can be described with`ADOP`+`FAMC`and an`ADOP`enum that says which spouse adopted (`HUSB`,`WIFE`,`BOTH`).The FamilySearch GEDCOM SpecifiÔÇŽ
- Association (laterals, e.g. godparents) uses`ASSO`with`ROLE`(e.g.`GODP`).The FamilySearch GEDCOM SpecifiÔÇŽ
- Documented extension tags are tied to meaning by a**URI**in`HEAD.SCHMA.TAG`, and semantic equivalence is by URI, not by the literal tag name.The FamilySearch GEDCOM SpecifiÔÇŽ
- GEDCOM explicitly allows alternative modeling choices (e.g., separation/reunion can be one family with multiple events vs multiple families with the same spouses).The FamilySearch GEDCOM SpecifiÔÇŽ

These are exactly the ÔÇťdomain states of affairsÔÇŁ you can turn into Paths, and then into`Iso`via`pathToIso`.
